# 컨텍스트 노트 — 작업 결정 기록

작업 중 내린 결정과 그 이유를 기록합니다.
새로운 결정이 생길 때마다 이 파일에 추가하세요.

## 2025 — 초기 설계 결정

### 배포 플랫폼: Netlify → Vercel로 변경

Netlify 무료 플랜의 크레딧 기반 과금 구조가 이 프로젝트의 사용 패턴과 맞지 않았습니다.
5초 polling 기준 월 392 크레딧이 필요한데 무료 플랜은 300 크레딧으로 초과합니다.
Vercel은 리소스 기반 과금이라 월 4회, 6시간, 20명 규모에서 무료 플랜으로 충분히 운영 가능합니다.

### 프레임워크: Vite React → Next.js 16 App Router로 변경

Vercel 선택 후 API Routes(Route Handlers)를 자연스럽게 쓰려면 Next.js가 가장 적합합니다.
별도 서버리스 함수 설정 없이 `src/app/api/` 경로만으로 엔드포인트를 구성할 수 있습니다.

### GAS 완전 제거 결정

초기에는 GAS를 API 서버로 유지하는 안을 검토했으나,
GAS의 `google.script.run` 방식은 외부 HTTP 호출과 호환되지 않아 `doGet`/`doPost` 전면 재작성이 필요했습니다.
재작성 공수가 크고 장기적으로 기술부채가 되므로 Vercel Route Handlers로 완전 대체하기로 결정했습니다.

### 실시간 동기화: Polling 30초 간격 채택

Google Sheets는 WebSocket을 지원하지 않습니다.
5초 polling은 Vercel 무료 플랜에서도 사용 가능하지만, 출석부 특성상 초단위 동기화가 불필요하고
30초 polling + Optimistic Update 조합으로 사용자 경험과 서버 비용 모두 만족할 수 있습니다.

Polling이 필요한 근본 원인은 **호스팅(Vercel)이 아니라 데이터 소스(Google Sheets)** 입니다.
Google Sheets API는 셀 변경을 push/webhook으로 알려주는 기능이 없고, Vercel의 서버리스 함수도
WebSocket 같은 영속 연결 유지에 적합하지 않습니다. 따라서 호스팅을 바꿔도 polling 자체는 사라지지 않습니다.
진짜 실시간 push가 필요하면 Supabase Realtime/Firebase 같은 DB로 교체해야 하지만,
그러면 "비개발자 교역자가 시트를 직접 열어 수정 가능"하다는 이 프로젝트의 핵심 장점을 잃게 되므로 채택하지 않습니다.

### 출석 상태 저장 방식: 레코드 유무로 판단

GAS 원본 코드 분석 결과, Attendance 시트에 `"출석"` 레코드가 존재하면 출석,
없으면 결석으로 간주하는 구조입니다. 별도 `"결석"` 값은 저장하지 않습니다.
마이그레이션 후에도 이 규칙을 동일하게 유지합니다 (기존 데이터 호환성).

### 관리자 인증: 세션 기반 간이 인증

별도 OAuth나 JWT 도입 없이 비밀번호 모달 → `POST /api/auth` 검증 → `sessionStorage` 토큰 저장 방식을 채택합니다.
공개 접근 서비스이고 관리자가 소수(교역자 수준)이므로 이 정도 보안 수준이 적합합니다.
비밀번호는 Vercel 환경변수(`ADMIN_PASSWORD`)에 보관합니다.

## Phase 1~2 — 프로젝트 세팅 및 출석체크 메인 UI 구현

### 마이그레이션 순서 변경: API 연동보다 UI 구현을 먼저 진행

원래 계획은 "Phase 2: API 연동 → Phase 3: 화면 구현"이었으나, 타입(`src/types/index.ts`)이 이미 확정되어 있어
목업 데이터로 먼저 화면을 만들고 나중에 실제 API로 교체하는 편이 더 효율적이라고 판단했습니다.
화면이 먼저 보여야 UX 문제를 일찍 발견할 수 있고, API 연동 후 수정 범위도 줄어듭니다.
이에 따라 진행 순서를 "UI 구현(목업) → API 연동 → UI-API 연결 → 배포"로 재조정했습니다.

### `create-next-app`은 빈 디렉토리가 아니면 실행 거부

`CLAUDE.md`, `docs/` 등 기존 파일이 있는 상태에서 `create-next-app .`을 실행하면 충돌로 거부됩니다.
임시 폴더(`tmpapp`)에 생성한 뒤 `src/`, `public/`, 설정 파일들만 골라 이동하고 임시 폴더를 삭제하는 방식으로 해결했습니다.
이 과정에서 `create-next-app`이 생성한 기본 `.gitignore`가 기존에 작성해둔 보안 강화 버전
(서비스 계정 키 패턴 `*-service-account.json`, `.vscode/*` 규칙 포함)을 덮어썼던 것을 발견해 원본으로 복원했습니다.

### 목업 데이터에는 절대 실제 캡쳐 속 이름을 쓰지 않음

UI 디자인 참고용으로 받은 스크린샷에는 실제 학생/교사 이름이 포함되어 있었습니다.
`docs/security-checklist.md` 정책에 따라 레이아웃/구조만 참고하고, 코드에 들어가는 목업 데이터는
스크린샷에 없는 더미 이름(홍길동, 김철수 등)으로 새로 작성했습니다 (`src/lib/mock-data.ts`).

### Next.js 16.2.9의 `next/font/google`에 한국어 서브셋 메타데이터 누락

`Hahmlet`, `Gowun_Batang`, `IBM_Plex_Sans_KR` 등 한글 폰트에 `subsets: ["korean"]`을 지정하면
"`korean`이 아니라 `latin`/`latin-ext`/`vietnamese`만 허용된다"는 타입 에러가 발생합니다.
이 Next.js 버전에 번들된 `next/font/google` 타입 정의 자체에 `korean` 서브셋이 어떤 폰트에도 존재하지 않는
버그성 누락으로 확인했습니다 (`node_modules/next/dist/compiled/@next/font/dist/google/index.d.ts` 확인).
해결: `subsets` 옵션을 생략하면 정상적으로 한글이 렌더링됩니다. 추후 Next.js 버전을 올리면 이 제약이 풀렸는지 재확인이 필요합니다.

### 출석체크 메인 UI 비주얼 컨셉: "주일 출석부" 티켓/스탬프 디자인

`frontend-design` 스킬로 메인 페이지를 리디자인하면서 출석 카드를 티켓 스텁(펀치홀, 도장 애니메이션) 모티프로 잡았습니다.
이후 사용자 피드백으로 색상을 빈티지한 크림톤에서 고등학생 타겟에 맞는 밝고 채도 높은 팔레트로 교체했고,
"산뜻함"과 상충되는 종이 그레인 텍스처는 제거했습니다. 티켓 펀치홀/스탬프 애니메이션 등 구조는 유지했습니다.
참고 캡쳐에 "출석 처리된 상태"의 색상 예시가 없어서, 학년/교사/새친구 구분 없이 출석 시에는 모두 동일한
스탬프 색(테라코타)으로 통일하기로 직접 결정했습니다 (그룹 구분은 평상시 테두리 색으로만 표시).

### 헤더 액션 버튼: 사용 빈도 기준으로 시각적 위계 재조정

초기 구현에서는 관리자 전용 저빈도 기능(`학생 관리`)이 가장 진하고 우측 끝(주 CTA 자리)에 있었고,
매주 쓰는 `출석현황`은 가장 옅은 버튼이라 실제 사용 빈도와 시각적 강조가 어긋났습니다.
`출석현황`을 채워진 배경으로 격상하고, `학생 관리`는 자물쇠 아이콘 + 구분선으로 "보기" 그룹과 분리했습니다.
`새로고침` 버튼은 30초 polling이 이미 자동 갱신하므로 제거하고 "마지막 업데이트 HH:MM" 텍스트로 대체했습니다
(날짜 표시 옆에 배치 — 액션 버튼과 성격이 달라 같은 줄에 섞으면 모바일에서 줄바꿈 시 정렬이 흐트러짐).

### 헤더 반응형: `flex-wrap` 즉흥 줄바꿈 대신 `lg` 기준 명시적 2단 레이아웃

화면 폭이 애매한 중간 구간(~800px)에서 `flex-wrap`이 컨텐츠 폭에 따라 즉흥적으로 줄을 바꾸면서
좌측 정렬된 콘텐츠 우측에 의도하지 않은 빈 공간이 생기는 문제가 있었습니다.
`lg`(1024px) 미만에서는 항상 날짜/세션 줄과 액션 버튼 줄을 각각 `w-full justify-between`으로 강제하고,
`lg` 이상에서만 한 줄로 합치도록 명시적으로 분기해 애매한 구간을 없앴습니다.

### 클라이언트 전용 타임스탬프는 `useState` 초기값이 아니라 `useEffect`에서 설정

"마지막 업데이트" 텍스트를 `useState(() => new Date()...)`로 초기화하면 서버 렌더링 시각과
클라이언트 하이드레이션 시각이 분 단위로 갈릴 때 React 하이드레이션 불일치가 발생할 수 있습니다.
`useEffect`로 마운트 후에만 값을 설정하도록 분리해 이 문제를 원천적으로 방지했습니다.

## Phase 3 — Google Sheets API 연동

### Students 시트의 `출석률` 컬럼은 헤더만 있고 데이터는 비어있음

실제 운영 스프레드시트 확인 결과 Students 시트에 `Notes` 뒤로 `출석률`/`세례`/`gender` 3개 컬럼이 있었습니다.
`세례`와 `gender`는 실제 값이 채워져 있지만, `출석률`은 헤더만 있고 모든 행이 빈 값입니다.
레거시 GAS 코드(`docs/legacy-gas.json`)에도 `출석률`을 Students 시트에 저장하는 로직은 없고
`getStudentAttendanceStats` 같은 함수로 Attendance 기록을 매번 스캔해 계산하던 것과 정황이 일치합니다 —
즉 이 컬럼은 실제로 쓰인 적 없는 placeholder로 추정됩니다.
타입(`src/types/index.ts`)에는 `attendanceRate`/`baptism`/`gender`를 모두 추가했지만,
Phase 3/4에서 `attendanceRate`는 항상 빈 문자열로 읽힐 것을 전제로 하고 별도 계산 로직을 만들지 않습니다.
이후 출석률 표시/계산이 실제로 필요해지면 레거시처럼 Attendance 기록 기반 계산 함수를 새로 만들어야 합니다.

### Attendance 시트의 `Ect` 컬럼은 `Type`의 오타가 아니라 별개의 비고 컬럼

처음에는 실제 운영 시트 I열 헤더가 `Type`의 오타로 `ect`가 된 것으로 보고 그렇게 기록했었으나,
`Ect` 컬럼의 실제 데이터 값(`타교회 선교(OO목사님 확인)` 등 출석 예외 사유 자유 텍스트)을 확인한 결과
`student`/`teacher` 구분과는 전혀 무관한 **별개의 기타 비고 컬럼**임이 드러나 정정합니다.
레거시 GAS 스크립트의 헤더 생성 코드는 분명히 9번째 컬럼을 `'Type'`으로 만들었지만(`appendRow([...,'Timestamp','Type'])`),
실제로 그 컬럼에 student/teacher 값을 쓰는 코드는 어디에도 없었습니다 — 즉 운영 중 누군가
그 자리에 `Type`이 아니라 `Ect`라는 별도 비고 컬럼을 만들어 쓰기 시작한 것으로 보입니다.

**결정**: `Ect` 컬럼은 그대로 유지하고 `AttendanceRecord.note`로 매핑합니다.
교사 출석 구분용 `Type`은 운영 시트에 존재하지 않으므로 Phase 3에서 **10번째 컬럼으로 신규 추가**해서
`AttendanceRecord.type`에 매핑합니다. `appendRow`로 행을 쓸 때는 9번째(`Ect`) 자리를 빈 문자열로 두고
10번째에 `type` 값을 씁니다 — `[date, session, grade, class, studentId, name, '출석', timestamp, '', type]`.

## Phase 7 사전 검증 — 검증용 사본 시트 준비 절차 (사본 교체 시 재실행)

실시트가 업데이트되어 검증용 사본을 새로 만들 때마다, 새 사본에는 검증에 필요한
변경 2가지를 다시 적용해야 합니다. 두 작업 모두 **멱등 스크립트**로 만들어져 있어
아래 순서대로 실행만 하면 됩니다 (이미 적용된 사본에 재실행해도 안전).

**사본 교체 절차:**

1. (수동) 실시트 → 사본 만들기 → 사본에 서비스 계정 이메일을 **편집자**로 공유
   - 사본이 공유 폴더에 생성되면 다른 사람 권한이 상속되므로 개인 위치로 이동 후 불필요한 권한 제거
2. (수동) `.env.local`의 `GOOGLE_SPREADSHEET_ID`를 새 사본 ID로 교체
3. `node scripts/verify-sheets.mjs` — 탭/헤더 구조가 코드 기대값과 일치하는지 확인
4. `node scripts/verify-data.mjs` — **Attendance에 `Type` 헤더 자동 추가**(없을 때만) + 데이터 형식 점검
5. `node scripts/migrate-newfamilies.mjs` — **NewFamilies 탭 → Students `Grade='새친구'` 마이그레이션**
   (ID 기준 중복 건너뜀, NewFamilies 탭은 보존, ID 없는 행은 제외 후 보고)
6. (선택) dev 서버 실행 후 `node scripts/verify-read-api.mjs` / `node scripts/verify-write-api.mjs`로 재검증

**참고 사항:**

- 검증 중 발견된 코드 버그 2건(stats 라우트의 레거시 행 교사 판별, 출석 현황 차트 라벨 밀림)은
  코드 쪽 수정이라 사본 교체와 무관하게 유지됩니다.
- 실시트에서 수동 수정이 권장되는 데이터 오류(Birthdate 오타, ID/Class 빈 값 등)는
  사본 교체 시 행 번호가 달라질 수 있으나, 4번 `verify-data.mjs`가 다시 검출해 줍니다.
  실시트에서 먼저 고치면 새 사본에는 문제가 아예 없습니다.
- Phase 7 실시트 전환 시에도 같은 절차의 4~5번(Type 헤더 + 새친구 마이그레이션)을
  실시트에 그대로 적용하면 됩니다 (Type 헤더는 Step 6 GAS 호환성 결론 확인 후).

## Phase 7 사전 검증 — 결과 및 결정 기록

실시트 사본으로 전 API·전 화면을 검증 완료했습니다. 세부 체크리스트와 수치는 CLAUDE.md 진행 상태 참조.
여기에는 결정 사항과 그 이유만 기록합니다.

### 새친구 데이터 모델: NewFamilies 탭 → Students `Grade='새친구'`로 통합

실제 운영에서 새친구는 Students가 아닌 `NewFamilies` 탭(19컬럼 별도 스키마)에 등록되고,
등반 시 Students로 이동(ID 유지)하는 구조였습니다. 새 앱은 처음부터 Students의 `Grade='새친구'` 행으로
설계했으므로, **NewFamilies를 Students로 마이그레이션하는 방식**을 채택했습니다 (`scripts/migrate-newfamilies.mjs`, 멱등).
잔여 필드(본래학년/등록일/인도자 등)는 Notes에 합칩니다. NewFamilies 탭은 레거시 GAS 병행 운영을 위해 보존합니다.
병행 기간 중 새로 등록되는 새친구는 한쪽에만 보이므로, 전환 직전에 스크립트를 재실행해 동기화합니다.

### stats 라우트의 교사 판별: Type 컬럼 + Grade='선생님' fallback

레거시 출석 행은 `Type`이 빈 값이고 `Grade='선생님'`으로 교사를 구분합니다(1,533건).
초기 코드는 `Type===''`을 학생으로 집계해 교사 통계가 0%로 나오는 버그가 있어,
`isTeacherRow`(Type='teacher' 또는 Grade='선생님') 판별로 수정했습니다.
전체 출석 합산도 재적 학년(gradeCount에 존재하는 학년)만 포함해, 명단에 없는 레거시 '새가족' 기록이
분모 없이 분자만 부풀리는 것을 방지했습니다.

### ID 생성 형식: 새 앱 코드 형식 유지 확정

실데이터 학생 ID 348건은 `YY-학년-반-순번(2자리)` 형식이지만, 최신 GAS의 `generateStudentId`는
새 앱과 동일한 `YYYY-학년-반-순번(3자리)`을 생성합니다 — 구형식은 과거 데이터일 뿐입니다.
연도 프리픽스가 달라 충돌이 불가능하고, 양쪽 어디에도 ID를 파싱하는 코드가 없어 형식 혼재는 무해합니다.
교사 ID도 같은 이유로 새 앱 형식(`T-YYYY-NNN`)을 유지합니다.

### 실시트 Type 컬럼 추가: GAS 호환 확인 완료

GAS `markAttendance`는 앞 8개 컬럼만 `appendRow`하고, 삭제/읽기는 전부 헤더 이름 기반이므로
Attendance 맨 끝(J열)에 `Type`을 추가해도 GAS는 영향받지 않습니다. GAS가 계속 쓰는 행은 Type이
비지만 위의 fallback으로 처리됩니다. (GAS는 6/3 내보내기 이후 수정 없음 — 사용자 확인)

### recharts 스택 막대의 LabelList는 index 대신 값으로 행을 찾아야 함

값이 0인 스택 세그먼트는 라벨 목록에서 빠져 `index`가 데이터 배열과 어긋납니다
(출석 현황에서 결석 0명인 팀이 있으면 이후 행 라벨이 한 칸씩 밀리던 버그의 원인).
`dataKey`를 카테고리명으로 주고 `value`로 행을 역조회하는 방식으로 수정했습니다 (`GroupAttendanceChart.tsx`).

### 반 정렬은 numeric localeCompare 필수

반 번호가 문자열이라 기본 localeCompare로는 10반이 1반과 2반 사이에 옵니다.
`localeCompare(..., { numeric: true })`로 수정 (`group-members.ts`, `birthdays.ts`).
테스트 시트에는 10반 이상이 없어 실데이터 검증에서만 드러난 문제였습니다.

### 실시트에서 수동 수정 필요한 데이터 (전환 전 권장)

행 번호는 검증 당시 사본 기준이며, `scripts/verify-data.mjs`가 재검출 가능:
Students 81행(Birthdate 하이픈 오타), 115행(ID 없음), 272행(Class 없음), NewFamilies 11행(ID 없음).
115·272행은 레거시 화면에서 아예 숨겨져 출석 체크가 불가능했던 학생들로, 새 앱 인원수(267)와
레거시(265)의 차이 원인이기도 합니다. 수정 시 양쪽 모두 267로 일치합니다.
